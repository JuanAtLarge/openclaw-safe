## Plugin Security

⚠️ **FINDING:** `plugins.allow` is empty

The warning appears in multiple openclaw commands:
```
plugins.allow is empty; discovered non-bundled plugins may auto-load: telegram
```

The Telegram plugin is the only one auto-loading and it's legitimate, but the empty `plugins.allow` means any plugin that happens to be discoverable could load without explicit approval.

**Recommendation:** Add to `~/.openclaw/openclaw.json`:
```json
"plugins": {
  "allow": ["telegram"]
}
```

### 4. `your-news-cron` 🟠 HIGH RISK

## Notable Architecture Observations

**Strong points:**
- All high-risk crons use `sessionTarget: "isolated"` — this is the most important mitigation and the user already has it
- The Reddit cron has a human review gate (the user approves before comments post)
- State files are kept in `~/projects/` not in workspace memory — good separation
- Only Telegram channel active, locked to pairing — small attack surface
- No ClawHub skills installed — avoids the 8% malicious skill problem entirely

**Concerns:**
- `your-news-cron` reads untrusted external content and posts to Twitter automatically with no human review gate
- `plugins.allow` is empty — should be locked down
- No memory file auditing/monitoring in place
- Telegram bot token is in the main config file — ensure `~/.openclaw/openclaw.json` is never committed to a public repo

## Overall Assessment

**this setup is reasonably secure for a personal/SMB setup.** The version is current, the biggest vulnerability (ClawJacked) is patched, and the use of isolated sessions for external-reading crons is the right call.

The main residual risk is prompt injection through the `your-news-cron` and `your-twitter-cron` crons, which read untrusted external content and post automatically. This is an accepted tradeoff for automation — it can be mitigated with tighter prompt constraints but not eliminated.

**Risk level: MODERATE** — no critical issues, a few improvements worth making.
